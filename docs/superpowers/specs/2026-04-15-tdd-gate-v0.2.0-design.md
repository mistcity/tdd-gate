# tdd-gate v0.2.0 设计文档

## 目标

集中做一波改进：修复体验短板 + 增加核心差异化功能（影响范围分析），为提交官方插件目录做准备。

## 一、体验短板修复

### 1.1 多目录测试文件支持

**问题**：当前只在同目录和 `__tests__/` 查找测试文件。很多项目把测试放在 `tests/`、`test/`、`spec/` 顶层目录。

**方案**：
- `file-classifier.ts` 的 `getExpectedTestPaths()` 扩展查找范围
- 对每个 impl 文件（如 `src/auth.ts`），除了查同目录和 `__tests__/`，还从项目根目录开始查找以下目录：
  - `tests/`、`test/`、`spec/`、`__tests__/`
- 匹配逻辑：basename 匹配（`auth.test.ts` 在 `tests/` 目录中 = 有效）
- config 新增 `testDirs` 字段，可自定义测试目录

**影响范围**：`file-classifier.ts`、`config.ts`、`types.ts`

### 1.2 MultiEdit 工具验证

**问题**：MultiEdit 路由到 `handleFileWrite()`，但 MultiEdit 的输入结构可能不含 `file_path`，导致静默放行。

**方案**：
- 查阅 Claude Code 的 MultiEdit 工具输入 schema
- 如果 MultiEdit 包含 `file_path`：当前代码正确，无需修改
- 如果不包含：从 MultiEdit 的 edits 数组中提取文件路径，逐个检查

**影响范围**：`pre-tool-use.ts`

### 1.3 自定义测试命令

**问题**：测试命令列表是硬编码的。用 `make test`、`./run_tests.sh` 等自定义命令的项目不会被 Stop 审计识别为"已跑测试"。

**方案**：
- config 新增 `testCommands` 字段（字符串数组）
- `bash-analyzer.ts` 的 `isTestCommand()` 合并用户自定义命令和内置列表
- 内置列表保持不变，用户列表是追加而非替换

**配置示例**：
```json
{
  "testCommands": ["make test", "./run_tests.sh", "bazel test"]
}
```

**影响范围**：`bash-analyzer.ts`、`config.ts`、`types.ts`

### 1.4 setup skill 增强

**问题**：当前 setup skill 只是一段静态说明文字。

**方案**：
- setup skill 改为指导 Claude 执行以下步骤：
  1. 检测项目中的测试框架（查找 `jest.config.*`、`vitest.config.*`、`pytest.ini`、`go.mod` 等）
  2. 检测测试目录布局（`tests/`、`__tests__/`、同目录等）
  3. 检测自定义测试命令（`package.json scripts.test`、`Makefile` 等）
  4. 生成推荐的 `tdd-gate.config.json`
  5. 输出当前项目的 TDD 覆盖情况摘要

**影响范围**：`skills/setup/SKILL.md`

## 二、差异化功能：影响范围分析

### 2.1 用户故事

Claude 改了 `auth.ts` 并跑了 `auth.test.ts`，想结束。tdd-gate 在 Stop hook 分析发现 `user-service.ts` 和 `api-handler.ts` import 了 `auth.ts`，它们的测试没有被执行。tdd-gate 拦截并告诉 Claude：

> "Impact analysis: you modified auth.ts. The following files depend on it and their tests were not run:
> - user-service.ts → run user-service.test.ts
> - api-handler.ts → run api-handler.test.ts
> Please run these tests before finishing."

Claude 去跑测试，tdd-gate 记录 TEST_RUN，下次 Stop 通过。

### 2.2 技术设计

**新模块**：`src/core/impact-analyzer.ts`

**流程**：
1. 接收 git diff 中的改动 impl 文件列表
2. 对每个改动文件，用 `grep -rl` 搜索项目中哪些文件 import/require 了它
3. 过滤掉测试文件和 exempt 文件，得到"受影响的 impl 文件"列表
4. 检查这些 impl 文件的测试是否在 journal 中有 TEST_RUN 或 TEST 记录
5. 返回未覆盖的文件列表

**Import 模式（10 种语言）**：

| 语言 | 正则模式 |
|------|---------|
| TypeScript/JavaScript/TSX/JSX | `import\s.*from\s+['"].*{basename}['"]` 和 `require\(['"].*{basename}['"]\)` |
| Python | `from\s+.*{module}\s+import` 和 `import\s+.*{module}` |
| Go | `import\s+.*["'].*/{package}["']` |
| Kotlin/Java | `import\s+.*\.{classname}` |
| Rust | `use\s+.*::{module}` 和 `mod\s+{module}` |
| C# | `using\s+.*\.{classname}` |

**性能保护**：
- 文件数量上限：最多 grep 500 个文件（超过跳过，允许通过）
- 超时保护：grep 操作总时间不超过 5 秒
- 可通过 config 关闭：`impactAnalysis: false`（默认 `true`）

**精度原则**：
- 不要求完美精度
- 宁可多拦（误报）也不漏拦（漏报）
- 用户觉得某个依赖不相关可以通过 config 加 exempt paths

### 2.3 Stop handler 修改

当前 Stop hook 流程：
1. `git diff` → 找改动文件
2. 分类 → impl 文件有没有对应测试被改/被跑
3. 没有 → deny

新增步骤（插入到步骤 2 和 3 之间）：
1. `git diff` → 找改动文件
2. 分类 → impl 文件检查
3. **影响范围分析** → 对每个改动的 impl 文件，找依赖方，检查依赖方的测试
4. 合并两步的未覆盖列表 → deny

**deny 消息区分两类**：
- 直接未覆盖：`"TDD violation: Write test file X first"`（现有行为）
- 影响范围未覆盖：`"Impact analysis: Y depends on modified Z, run Y.test.ts"`（新增）

### 2.4 配置扩展

```json
{
  "impactAnalysis": true,
  "impactAnalysisMaxFiles": 500,
  "impactAnalysisTimeout": 5000,
  "testCommands": [],
  "testDirs": ["tests", "test", "spec", "__tests__"]
}
```
